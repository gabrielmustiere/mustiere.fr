The JavaScript runtime wars have lost their religious edge. All three options hold up in production, their audiences have clarified, and the debate is no longer "which one survives" but "which one hurts least for this case."

## What each has consolidated

**Node**: the full npm ecosystem, predictable LTS support, the best commercial observability tooling, and for a few versions now a native test runner and watcher. It's the option no recruiter will question.

**Bun**: impressive startup times, a package manager 10× faster, a bundler, a test runner and a TS transpiler built in. The tooling argument is real. Rough edges remain on less popular Node APIs and some native modules.

**Deno**: security-by-default philosophy, excellent native TypeScript support, a runtime designed for the edge (Deno Deploy). The ecosystem is smaller, but URL-based imports have matured.

## My current heuristic

- **Existing project with a lockfile**: stay on Node. The migration gain doesn't cover the risk.
- **New project, team of 1–5**: Bun. Integrated tooling pays back the learning curve quickly.
- **Scripts, short workers, edge functions**: Deno, especially if you're targeting Deno Deploy.
- **Ultra-critical cold-start latency**: Bun or Deno, depending on the host.

The debate is no longer existential. It's a tooling choice — to be made with the same calm as picking a linter.
