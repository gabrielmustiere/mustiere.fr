Il y a un moment, dans l'adoption des LLMs en production, où l'équipe se rend compte qu'elle vole à l'aveugle. Les démos impressionnent, les tickets rentrent, et personne ne sait vraiment si on *régresse* à chaque changement de prompt ou de modèle. C'est à ce moment-là qu'on décide d'écrire une suite d'évaluations.

Dans 9 cas sur 10, cette suite finit par ne rien évaluer du tout.

## Le piège du benchmark public

La tentation initiale est d'aller chercher un benchmark qui existe déjà — MMLU, GSM8K, le dernier dataset en vogue — et de reporter le score. C'est confortable, c'est objectif, c'est comparable. Le problème : ça ne mesure pas **votre** produit.

Votre produit a une distribution d'entrées spécifique, un domaine métier, des contraintes de format, un contexte qui pèse beaucoup dans la sortie. Aucun benchmark public ne capture ces trois choses.

> Un modèle qui gagne 2 points sur MMLU peut perdre 15 points sur votre cas. L'inverse est aussi vrai.

## Partir des cas d'échec, pas des cas idéaux

La seconde erreur consiste à écrire, le vendredi après-midi, une trentaine d'exemples « jolis » qui correspondent au happy path. Ils passent, tout va bien, on déploie. La production, elle, ne passe pas par le happy path.

L'approche qui tient : dès le premier jour en prod, instrumenter les logs, et retenir tous les cas où **un humain a dû reprendre la sortie**. Ces cas deviennent la *graine* de la suite d'éval. On les anonymise, on les fige, on les exécute à chaque changement.

### Petit harnais minimal

Un script suffit. Voici une version TypeScript, sans dépendance :

```ts
// eval-harness.ts — tient en < 50 lignes
type Case = { id: string; input: string; expected: string };
type Result = { id: string; score: number; output: string };

async function* runSuite(
  suite: Case[],
  model: (input: string) => Promise<string>,
  score: (out: string, exp: string) => number,
): AsyncGenerator<Result> {
  for (const c of suite) {
    const out = await model(c.input);
    yield { id: c.id, score: score(out, c.expected), output: out };
  }
}

const suite = await loadFrom('./cases/*.json');
const results: Result[] = [];
for await (const r of runSuite(suite, gpt, semantic)) {
  results.push(r);
  console.log(r.id, r.score.toFixed(2));
}
const mean = results.reduce((a, r) => a + r.score, 0) / results.length;
console.log(`mean = ${mean.toFixed(3)}`);
```

La fonction `score` est le vrai sujet. Dans mon expérience, vous aurez besoin d'un mélange :

- Des règles dures pour le format (JSON valide, longueur max, champs obligatoires).
- Des règles lexicales pour le vocabulaire métier (synonymes acceptés, interdictions).
- Un juge LLM pour la *qualité sémantique*, calibré sur quelques centaines d'exemples annotés à la main.

## La métrique qui compte vraiment

Après six mois d'utilisation, la seule métrique qui prédit correctement la satisfaction utilisateur est le *taux de reprise humaine* : sur 100 sorties, combien un opérateur a dû en modifier pour qu'elles soient acceptables ? C'est brut, c'est coûteux à instrumenter, c'est la seule qui tient.

Tout le reste — scores sémantiques, BLEU, ROUGE, LLM-as-judge — n'est qu'une **approximation** de ce signal. Utile en CI rapide, inutile en l'absence du signal primaire.

## Ce qu'il ne faut pas faire

- Mesurer uniquement sur du golden data écrit par l'équipe.
- Ne pas versionner la suite avec le code.
- Faire confiance à un score global sans regarder la distribution par segment.
- Remplacer le juge humain par LLM-as-judge sans calibration.

L'évaluation LLM n'est pas un sous-problème du ML. C'est un problème produit : vous devez savoir ce que vaut une sortie *pour votre utilisateur*. Tant que vous n'avez pas ça, vous n'évaluez rien.
