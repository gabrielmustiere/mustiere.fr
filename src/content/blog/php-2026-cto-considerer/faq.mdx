questions:
  - q: 'Faut-il vraiment démarrer un nouveau projet en PHP/Symfony plutôt qu’en Node ou Python en 2026 ?'
    r: |
      Ça dépend du produit. Pour une app web qui consomme des LLM via API et qui a besoin d’une couverture API + back-office + e-commerce + mobile,
      Symfony est aujourd’hui le choix le plus rapide à mettre en route — API Platform, EasyAdmin, Sylius, Live Components et le bridge Hotwire Native
      sortent tout de la boîte, sans glue code. Pour un produit fortement orienté ML (entraînement, inférence custom, traitement de données massif),
      Python reste mieux placé. Pour un service très realtime sub-milliseconde ou du tooling système, Go ou Rust tirent leur épingle du jeu. Le piège est
      de raisonner par genre plutôt que par cas d’usage.
  - q: 'FrankenPHP en worker mode, quel coût de migration sur une codebase Symfony existante ?'
    r: |
      Faible, dans la majorité des cas. La règle tient en une phrase : ne pas conserver d’état mutable entre deux requêtes. Les rares services qui
      cachent de la donnée request-scoped doivent implémenter `ResetInterface`, ce qui se corrige rapidement. Une fois cette étape passée, le gain est
      immédiat : sur le bench Symfony 7.4 publié début 2026, on passe de 1 240 RPS / p95 45 ms (Nginx + PHP-FPM) à 3 850 RPS / p95 8 ms (FrankenPHP
      worker). Le déploiement se simplifie aussi — binaire unique, Caddy intégré, HTTPS auto, HTTP/3 et Mercure natifs. RoadRunner et Swoole restent des
      alternatives sérieuses pour qui a déjà investi dessus.
  - q: 'Combien de temps pour qu’une équipe Node ou Python devienne productive en Symfony ?'
    r: |
      Sur les équipes que j’ai vu basculer, un développeur sénior à l’aise avec un framework full-stack typé (Spring, Rails, NestJS, Django) atteint un
      niveau correct en deux à quatre semaines, et la productivité maximale en deux à trois mois. Le frein principal n’est pas le langage — PHP 8.5 est
      proche d’un C# moderne ou d’un Kotlin — mais la familiarisation avec les conventions Symfony (DI, services, Doctrine, Messenger). C’est un
      investissement modeste comparé à la couverture fonctionnelle qu’on récupère en retour.
